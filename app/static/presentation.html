Motivation

Why Not Just Use Yelp?

Traditional restaurant search platforms have a fundamental limitation: they rely on rigid metadata, not on what customers actually say.

❌ Yelp's Approach

  • Rigid checkboxes: "Italian", "$$", "Open Now"
  • Keyword search only matches exact words
  • Can't express atmosphere or feeling
  • Same results for "date night" and "birthday dinner"

✅ Our Approach

  • Natural language: type anything you want
  • AI reads review text to understand meaning
  • Understands vibes, moods, and occasions
  • Different results for different intents

Searching "loud sports bar with cheap wings" on Yelp requires checking 3+ boxes. In our app, you just type it.

Price Imputation

Yelp's price column is often missing. We wrote a custom function (extract_price_from_reviews) that scans review text for price cues like "cheap", "expensive", or "$30 entrees" and infers the price tier.

Review Aggregation

Each restaurant's top reviews (2019+) are concatenated into a single "profile document." This captures the collective voice of real diners — their opinions on food, service, noise, and ambiance.

Core Technology 1 — Week 4

Semantic Vector Search

We use a pre-trained language model to convert text into dense numerical vectors, enabling mathematical similarity comparison.

How It Works

1. Each restaurant's review profile is passed through SentenceTransformer (MiniLM) → a 384-dimensional vector.

2. The user's query is embedded using the same model → another 384-D vector.

3. We compute the Dot Product between the query vector and every restaurant vector. The highest scores = the most relevant restaurants.

4. Because all vectors are L2-normalized, Dot Product equals Cosine Similarity.

similarity(query, restaurant) = q⃗ · r⃗

For L2-normalized vectors: cos(θ) = q⃗ · r⃗

The vector for "budget friendly brunch" is mathematically close to vectors of restaurants whose reviews mention "cheap", "affordable", "weekend brunch". No keyword matching needed.

Core Technology 2 — Algorithm From Scratch (Week 7)

Custom K-Means Clustering

To satisfy the from-scratch algorithm requirement, we implemented a fully vectorized K-Means using only NumPy — no scikit-learn.

The Algorithm (src/kmeans.py)

# Pure NumPy — no sklearn
def kmeans(X, k=15, max_iter=100):
  centroids = X[random_indices]
  for _ in range(max_iter):
    dists = X @ centroids.T
    labels = np.argmax(dists, axis=1)
    for j in range(k):
      centroids[j] = mean(X[labels==j])
  return labels, centroids

What It Produces

All restaurants are grouped into 15 "Vibe Clusters". We analyzed each cluster's top restaurants and assigned human-readable labels:

  • Cafes & Coffee — cozy study spots, latte art
  • Pizza & Casual Italian — BYOB, red sauce
  • Asian Fusion & Chinese — Chinatown favorites
  • Casual Pubs & Sports Bars — wings, beer, TVs

Diversity Penalty

If the top 3 results belong to the same cluster, later entries from that cluster receive a -10% penalty so other vibes can surface.

Core Technology 3

Multi-Signal Reranking Engine

Pure semantic similarity can surface relevant-but-low-quality places. Our reranker blends 5 weighted signals into a single final score.

40%
Location
20%
Star Rating
15%
Price Match
15%
Popularity
10%

Bayesian Average

We don't trust raw star ratings. A 5.0★ with 1 review is inflated. Our Bayesian Average anchors all ratings toward a global mean of 3.5 with a confidence threshold of 50 reviews: B = (n·s + m·C)/(n+m)

Log Popularity

Review counts are log-normalized against an absolute ceiling of 3,000 reviews: log(1+count)/log(1+3000). This prevents a restaurant with 5,000 reviews from completely dominating one with 500.

Intelligence Layer

Dynamic Weight Adjustment

The reranker doesn't use static weights — it analyzes the user's query to detect intent and shifts weights accordingly.

📊 Popularity Intent

Keywords: "most reviews", "popular", "most reviewed"

→ Popularity weight jumps to 50%, similarity drops to 15%

⭐ Rating Intent

Keywords: "highest rated", "best rated", "top rated"

→ Star rating weight jumps to 50%

💰 Price Intent

Keywords: "cheap", "budget", "expensive", "pricey"

→ Activates price preference scoring via regex keyword matching

🧠 Smart Weight Reallocation

If a signal is unused (e.g., no location mentioned, no price preference), its weight is automatically redistributed to boost the semantic similarity signal. This prevents unused signals from diluting the overall Match Score.

Query: "cafe with the most reviews" → The system detects "most reviews" as a popularity intent, shifts 50% weight to review count, and returns restaurants sorted primarily by how many people have reviewed them — while still remaining semantically relevant to "cafe".

System Design

Application Architecture

We moved beyond Streamlit to build a production-grade web application with a clear separation between backend logic and frontend presentation.

⚡

FastAPI Backend

A high-performance Python API server (app/api.py) that loads embeddings into memory once at startup and serves recommendations via a GET /api/search endpoint. Supports async requests and auto-reloading.

🎨

Custom Frontend

Vanilla HTML/CSS/JS with a dark-mode glassmorphism design. Client-side filtering (price, stars, neighborhood) runs instantly in the browser without server round-trips. Results paginate 10 at a time.

🗺️

Smart Neighborhood

When a user selects "Center City" from the dropdown, the frontend appends "in Center City" to the semantic query. The AI finds restaurants in that area even when Yelp's neighborhood data is missing.
